# Dry run на собственных реальных фотографиях — 14 июля 2026 года

**Статус:** технический открытый dry run; не валидация точности и не основание экспертного вывода.  
**Оператор и владелец изображений:** Смагулов Амирхан Каиржанович.  
**Рабочая пара:** Community Forensics 384 и SPAI.  
**Ground truth:** `real`, основание — контролируемая съёмка владельцем на Apple iPhone 15.

## 1. Подготовка набора

- локально получено 27 исходных файлов HEIC без автоматической конвертации Windows;
- SHA-256 каждого файла после копирования совпал с SHA-256 источника;
- для development выбраны первые 20 файлов: `IMG_9505.HEIC`—`IMG_9524.HEIC`;
- исходные HEIC сохранены без изменений и не передавались детекторам;
- для каждого из 20 выбранных оригиналов создан контролируемый JPEG;
- параметры производной копии: ImageMagick 7.1.2-26, primary frame `[0]`, `auto-orient`, JPEG quality 95, sampling factor 4:4:4;
- в JPEG сохранились EXIF с моделью `iPhone 15` и ICC-профиль;
- реестр содержит 20 семейств, 20 HEIC-оригиналов и 20 JPEG-производных; битых parent-ссылок и повторяющихся SHA-256 нет;
- фотографии и полный локальный реестр исключены из Git и не загружены в Google Drive.

Локальный набор: `work/datasets/iphone-202607-real/`.  
Объединённые обезличенные результаты: `REAL_DRY_RUN_RESULTS_2026-07-14.csv`.

## 2. Среда запуска

- GPU: NVIDIA GeForce RTX 5050, 8 GB VRAM;
- драйвер NVIDIA: 610.62;
- PyTorch: 2.12.1+cu132;
- CUDA внутри PyTorch: доступна;
- Community Forensics: профиль 384;
- SPAI: официальный вес `spai.pth`, вход ограничен по длинной стороне до 1024 px, batch size 1, два data workers.

Первый полноразмерный запуск SPAI был остановлен как эксплуатационно неразумный: восемь workers заняли около 7.6 GB VRAM и до 14.5 GB общей GPU-памяти, а RAM компьютера достигла 91%. Это не ошибка исходных данных. Рабочий wrapper изменён: по умолчанию `ResizeTo=1024` и `DataWorkers=2`. После изменения 20 изображений обработаны примерно за 30 секунд, память GPU освободилась штатно.

## 4. Вывод dry run

1. Оба детектора технически работают на RTX 5050 и выдают результаты на собственных фотографиях.
2. Даже на документированных реальных снимках встречаются высокие ложноположительные scores.
3. Модели ошибаются на разных файлах; голосование или правило «хотя бы один выше 0.5» резко увеличило бы число ложных сигналов.
4. Dry run прямо поддерживает запрет score-only вывода и обязательную категорию `Hu` при конфликте.
5. Полученные 5% и 30% нельзя выдавать за оценку точности: выборка мала, открыта и использована для отладки процесса.
6. Пороги по этой выборке не подбираются. Иначе development-набор начнёт подменять закрытую валидацию.
